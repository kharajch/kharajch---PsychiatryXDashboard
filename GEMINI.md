# PsychiatryX Dashboard: Project Mandates

This document defines the foundational mandates and architectural constraints for the PsychiatryX Dashboard. All development MUST strictly adhere to these rules.

## 1. Hybrid Architecture Integrity
-   **Dual-Client Parity:** Any new feature added to the "Cloud" Next.js client MUST be evaluated for inclusion in the "Local" standalone HTML client (`PsychiatryX_Dashboard.html`).
-   **Offline-First:** All clinical data entry workflows MUST prioritize offline persistence via RxDB. Synchronization is automated and zero-config for the user.
-   **Direct HTML Maintenance:** The standalone HTML file is a first-class citizen. It uses origin-aware detection to connect to the backend sync API without manual server URL entry.

## 2. Data & Sync Standards
-   **Schema Consistency:** Models defined in `lib/models` (Mongoose) MUST remain synchronized with RxDB schemas in both the HTML client and the Next.js frontend. Standardized schema values and formats (e.g., gender `"Non-Binary"` and ID prefix `"MKS-"`) must be strictly maintained across all clients.
-   **RxDB REST Replication:** All synchronization must flow through the unified API endpoints at `/api/sync/[collection]`.
-   **Automated Connectivity:** The local client MUST automatically target the correct backend (Production Vercel or Localhost) based on the environment. On HTTP origins it targets `window.location.origin`; on `file://` protocols, it probes `http://localhost:3000` (600ms timeout) before falling back to Vercel production.
-   **No Partial Sync:** Ensure that clinical assessments are synced atomically to prevent inconsistent patient states.
-   **Safe Tombstone Processing:** Replication handlers MUST guard property modifications. During push replication, check if metadata fields (like `_deleted`) exist before resetting them, preserving values transformed by the push modifier.

## 3. Clinical & Safety Rules
-   **High-Risk Flagging:** Never modify assessment scoring logic without validating against clinical standards (e.g., DSM-5 / ICD-11 markers).
-   **Suicide Risk Visibility:** SRA-20 and critical depression markers MUST always trigger immediate visual alerts in the UI.
-   **Audit Logging:** Every modification to patient data MUST be logged via the `AuditLog` model to maintain medical record integrity.

## 4. UI/UX Standards
-   **Modular Subcomponent Layout:** Frontend code must be clean and modular. Extract components from `app/page.tsx` into `app/components/` and use individual CSS modules (`*.module.css`) to enforce scoped, clean styles.
-   **Stitch Design System:** Adhere to the modern "Crimson & Noir" dark theme (crimson red as primary, black as secondary) using Google Stitch design tokens. Immersive, fluid particle canvas background elements (Three.js/R3F) and layout micro-animations (Framer Motion/GSAP) are integrated. All Next.js and HTML client code must use Vanilla CSS for styling; Tailwind CSS is strictly prohibited.
-   **Accessibility:** Ensure keyboard-first navigation for rapid data entry (e.g., mapping numerical keys to assessment options).
-   **PDF Consistency:** PDF reports generated by the HTML client and Next.js client MUST have identical visual layouts and branding.
-   **Mobile Responsiveness:** Layouts must dynamically adjust to mobile viewports. Hardcoded inline grid styles must be avoided in favor of CSS media-query-aware grid classes, and navigation must adapt to a touch-friendly collapsible drawer with overlay backdrop.
-   **Modal Form Submission Integrity:** Asynchronous modal form submissions MUST read DOM values synchronously prior to executing any async tasks. This prevents race conditions when the modal is closed concurrently.
-   **Immersive Authentication:** The login and registration experience MUST be perfectly centered using the `.auth-overlay` Flexbox pattern with a `backdrop-filter: blur(8px)` to ensure professional clinical focus.
-   **Dynamic Staggering:** All primary overview components (dashboard cards, navigation items) MUST utilize staggered entrance animations to improve perceived performance and interactivity.
-   **E2E Motion Snapping:** Any layout component or card utilizing transitions or animations (Framer Motion, CSS, or GSAP) MUST conditionally bypass all motion/delay attributes and snap to their target values instantly in zero seconds during Playwright/automated testing.
    - The `isTestEnv` state MUST be resolved lazily in a `useEffect` on mount browser-side to prevent server-side rendering (SSR) mismatch or page load shifts.
    - Automated tests apply the `body.no-animations` class which resets both `animation: none !important;` and `transition: none !important;`.

## 5. Security & Privacy
-   **Zero Leakage:** Never log or print Patient Identifiable Information (PII) in console logs or error reports.
- **Clinician Registration & Auth:** Secure clinician registration (`/api/auth/register`) creates new isolated clinic environments and returns signed NextAuth JWT tokens for instant login. NextAuth and token authentication (`/api/auth/login`) are supported for cross-device access with a fallback. **CRITICAL:** For production deployments with real patient data, `NEXTAUTH_SECRET` MUST be configured to enforce strict authentication and disable fallback. For development environments, the system provides a temporary fallback secret to prevent 500 errors during testing if the variable is missing.

-   **Data Sanitization:** All user-provided clinical notes must be sanitized before being rendered or stored.
-   **Database Credentials Configuration:** To prevent database connection errors, ensure `MONGODB_URI` has its password special characters correctly URL-encoded (e.g., `%40` instead of `@`). MONGODB_URI presence checks must occur within connection routines to prevent build-time crashes.
